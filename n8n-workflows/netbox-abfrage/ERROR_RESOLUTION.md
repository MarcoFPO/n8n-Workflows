# N8N Deployment - Error Resolution Guide

## Problem: "additional properties" Error

### Error Message
```json
{
  "message": "request/body must NOT have additional properties"
}
```

### Root Cause Analysis

The n8n API uses strict JSON Schema validation for the `/api/v1/workflows` POST endpoint. The schema defines exactly which fields are allowed in the request body.

**Fields NOT allowed in request body:**
- `description` - Should be a separate field or omitted
- `id` - Generated by API
- `active` - Read-only, cannot be set via API
- `versionId` - Generated by API
- `createdAt` / `updatedAt` - Auto-generated
- Any other undeclared properties

**Fields REQUIRED in request body:**
- `name` - Workflow name
- `nodes` - Array of workflow nodes
- `connections` - Object defining node connections
- `settings` - Workflow configuration object

### Original File Structure (WRONG)
```json
{
  "name": "Test: NetBox-Abfrage Sub-Workflow",
  "description": "Test-Workflow für...",  ← NOT ALLOWED
  "settings": {
    "executionOrder": "v1"
  },
  "nodes": [...],
  "connections": {...}
}
```

### Fixed File Structure (CORRECT)
```json
{
  "name": "Test: NetBox-Abfrage Sub-Workflow",
  "settings": {
    "executionOrder": "v1"
  },
  "nodes": [...],
  "connections": {...}
}
```

## Solution Steps

### Step 1: Remove Disallowed Fields
Remove all fields not in the API schema:
- ❌ `description`
- ❌ `id`
- ❌ `active`
- ❌ `versionId`
- ❌ `createdAt`
- ❌ `updatedAt`
- ❌ etc.

### Step 2: Ensure Required Fields
Verify the request includes:
- ✅ `name` (string)
- ✅ `nodes` (array)
- ✅ `connections` (object)
- ✅ `settings` (object with executionOrder)

### Step 3: Verify Node Structure
Each node must have:
```json
{
  "parameters": {},
  "id": "unique-id",
  "name": "Node Name",
  "type": "n8n-nodes-base.nodeType",
  "typeVersion": 1,
  "position": [x, y]
}
```

### Step 4: Verify Connection Structure
Connections must follow this pattern:
```json
{
  "connections": {
    "NodeNameA": {
      "main": [
        [
          {
            "node": "NodeNameB",
            "type": "main",
            "index": 0
          }
        ]
      ]
    }
  }
}
```

## API Schema Validation

### POST /api/v1/workflows
**Allowed Fields:**
```
{
  "name": string (required),
  "nodes": array (required),
  "connections": object (required),
  "settings": {
    "executionOrder": string (required),
    ... other optional settings
  },
  "meta": object (optional),
  "pinData": object (optional),
  "staticData": object (optional)
}
```

**NOT Allowed:**
- `id` - Auto-generated by API
- `active` - Set after deployment with activate endpoint
- `versionId` - Auto-generated
- `versionCounter` - Auto-generated
- `description` - Use meta field instead if needed
- `createdAt` / `updatedAt` - Auto-generated
- `triggerCount` - Auto-generated
- `activeVersionId` - Auto-generated
- `isArchived` - Use separate API endpoint
- Any undeclared properties

## How n8n Handles Description

In n8n, workflow descriptions should be handled differently:

### Option 1: Store in Meta
```json
{
  "name": "Workflow Name",
  "meta": {
    "description": "Long description here"
  },
  "nodes": [...],
  "connections": {...},
  "settings": {...}
}
```

### Option 2: Use Web UI
Deploy workflow via API, then update description in the web interface.

### Option 3: Use PUT Endpoint After Creation
```bash
PUT /api/v1/workflows/{id}
{
  "name": "Workflow Name",
  "description": "...",  # This field is allowed in PUT
  "nodes": [...],
  "connections": {...},
  "settings": {...}
}
```

## Common Mistakes to Avoid

1. ❌ Including `description` in POST body
   - **Fix:** Remove it or store in `meta` object

2. ❌ Trying to set `active: true` in creation request
   - **Fix:** Create workflow first, then use activate endpoint

3. ❌ Using `PUT` instead of `POST` for new workflows
   - **Fix:** Use POST for creation, PUT for updates

4. ❌ Forgetting `settings.executionOrder`
   - **Fix:** Always include: `"settings": {"executionOrder": "v1"}`

5. ❌ Node names in connections don't match node names in array
   - **Fix:** Ensure exact string match between `node.name` and connections key

6. ❌ Invalid node positions
   - **Fix:** Position should be array: `"position": [x, y]` where x, y are numbers

7. ❌ Missing required node properties
   - **Fix:** Every node needs: id, name, type, typeVersion, parameters, position

